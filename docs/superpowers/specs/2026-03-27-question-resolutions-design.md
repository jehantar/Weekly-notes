# Open Question Resolutions

## Problem

Open questions in the weekly analysis are generated by the LLM but have no way to be marked as resolved. The user wants to track which questions have been answered, optionally record the resolution, and have this state survive analysis regeneration.

## Design

### Data Model

New `question_resolutions` table in Supabase:

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | `gen_random_uuid()` |
| user_id | uuid FK → auth.users | NOT NULL |
| week_start | date | NOT NULL |
| question_hash | text | First 16 hex chars of SHA-256 of normalized question text. NOT NULL |
| question_text | text | Original question for display. NOT NULL |
| resolution | text | Optional answer/context the user provides |
| resolved_at | timestamptz | When resolved, defaults to `now()` |

**Constraints:** `UNIQUE(user_id, week_start, question_hash)`

**RLS:** Users can only CRUD their own rows (`auth.uid()::text = user_id::text`).

### Hashing Strategy

`hashQuestion(text: string): string` in `src/lib/utils/strings.ts`:

1. Lowercase the text
2. Trim whitespace
3. Strip trailing punctuation (`?`, `.`, `!`)
4. SHA-256 via Web Crypto API (`crypto.subtle.digest`)
5. Take first 16 hex characters

This allows matching LLM-generated questions against stored resolutions even after regeneration. Minor rephrasing by the LLM would cause a mismatch — the question reappears as open, which is acceptable.

The function is async (Web Crypto requirement). Hash results are computed once per render cycle and memoized.

### API Route

`POST /api/question-resolutions` — single route, two actions:

**Resolve:**
```json
{
  "action": "resolve",
  "weekStart": "2026-03-24",
  "questionText": "Who is the GPSL for GCS?",
  "resolution": "Sarah confirmed it's Mike Chen"
}
```
- Server computes `question_hash` from `questionText`
- Upserts row into `question_resolutions`
- Returns the created resolution

**Unresolve:**
```json
{
  "action": "unresolve",
  "weekStart": "2026-03-24",
  "questionHash": "a1b2c3d4e5f67890"
}
```
- Deletes the matching row
- Returns success

Auth: cookie-based via `createClient()` from `@/lib/supabase/server` (consistent with all existing routes).

### Data Loading

In the server component (`page.tsx`), fetch resolutions in parallel with the existing week data queries:

```typescript
supabase
  .from("question_resolutions")
  .select("*")
  .eq("user_id", user.id)
  .eq("week_start", weekStart)
```

Pass results into `WeekProvider` as initial state.

### State Management

`WeekProvider` additions:

- **State:** `questionResolutions: Map<string, QuestionResolution>` keyed by `question_hash`
- **Actions:**
  - `resolveQuestion(questionText: string, resolution?: string)` — optimistic update, then API call
  - `unresolveQuestion(questionHash: string)` — optimistic update, then API call, roll back on failure

No new context provider needed — fits in existing `WeekProvider`.

### UI

**Open Questions section** (`analysis-sections.tsx`):

The component receives the LLM-generated `openQuestions` array and the `questionResolutions` map. It hashes each question and splits into two lists:

1. **OPEN QUESTIONS** — questions with no matching resolution
   - Each row has a resolve button (checkmark) on the right, visible on hover
   - Click resolves optimistically, shows toast "Question resolved" with Undo link (existing delayed-delete pattern)

2. **RESOLVED (n)** — collapsible section below, defaulting to collapsed
   - Each row shows: original question (dimmed), resolution text if provided, reopen button on hover
   - When a question is first resolved, a text input appears for that item with placeholder "Add resolution (optional)..." — submit on Enter, collapses on blur

### Regeneration Behavior

When the user regenerates the analysis:
- The LLM produces a fresh `openQuestions` array
- On render, each question is hashed and checked against stored resolutions
- Previously resolved questions automatically land in the Resolved section
- New questions from the LLM appear in Open Questions
- Resolutions for questions the LLM no longer generates remain in the DB but are invisible (no matching question to display them against)
